## What this project is

A GitHub Pages-hosted digital signage system for Barco ClickShare Base Units. Each Base Unit is configured to point at a URL that serves an image slideshow. Marketing manages content by uploading images to GitHub — no IT involvement needed for content updates.

**Hosting:** GitHub Pages (public repo required for free tier)
**Delivery:** Static files served via GitHub's CDN
**Content updates:** Upload images → GitHub Actions generates manifest → Pages deploys → display updates automatically

## Architecture

```
clickshare-signageV2/
├── .github/
│   └── workflows/
│       └── update-manifests.yaml
├── scripts/
│   └── generate_manifests.py
├── units/
│   └── [unit-name]/
│       ├── index.html          ← identical across all units
│       ├── images.json         ← auto-generated, do not edit manually
│       └── images/
│           └── .gitkeep
├── .nojekyll
└── README.md
```

**Key structural decisions:**
- Each unit lives in `units/[unit-name]/`
- `index.html` is per-unit but identical — fetch path `images.json` is relative, so each unit loads its own manifest automatically
- `images.json` is auto-generated by GitHub Actions — only `interval` and `transition` fields are safe to edit manually
- `.nojekyll` at repo root — skips Jekyll processing, direct file serving only
- Python script is external (`scripts/generate_manifests.py`), not inline in the workflow YAML — keeps it testable and diffable locally

## GitHub Actions workflow

**Trigger:** Push to `main` that modifies any file under `units/**/images/`

**Behavior:**
- Discovers all unit folders that have an `images/` subdirectory
- Scans each `images/` folder for supported extensions: `.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`
- Loads existing `images.json` to preserve `interval` and `transition` if already set
- Writes new `images.json` sorted alphabetically by filename
- Commits changes with `[skip ci]` tag to prevent re-triggering
- Only commits if manifest actually changed (idempotent)

**Permissions:** Declared at the job level, not workflow level (`contents: write`)

**Timeout:** 5 minutes on the job

**Action pinning:** `actions/checkout` should be pinned to a full commit SHA, not a floating tag

**Default timing (baked into `generate_manifests.py`):**
- `interval`: 15000ms (15 seconds per slide)
- `transition`: 2000ms (2 second crossfade)

## Slideshow behavior (index.html)

**Loading indicator:**
- `#loading` div is hidden manually via JS after the first image's `load` event fires — not via CSS selector on DOM mutation
- Check `img.complete` before attaching `onload` to handle cached images correctly
- Only the first slide triggers this — remaining slides load in the background at `opacity: 0`

**Slide rotation:**
- `setInterval` cycles through slides at `interval` ms
- CSS `transition-property: opacity` + `transition-timing-function: ease-in-out`
- `transition-duration` set per-element via JS from config

**Hybrid auto-refresh watcher (built in from the start):**
- A second `setInterval` runs independently alongside the rotation timer
- Every 2 minutes: fetches `images.json?t=<timestamp>` (cache-busted)
- Compares `JSON.stringify(fresh.images)` to `JSON.stringify(loadedImages)`
- If different: calls `location.reload()` — browser re-renders with new content, images already cached so reload is fast (~1 second black flash)
- If same: does nothing, no interruption to display
- If fetch fails: logs error, continues running current slides, retries next interval
- `loadedImages` captured at init time and passed into the watcher

**End-to-end update timing:** ~4 minutes worst case (2 min pipeline + 2 min poll)

## ClickShare integration

Each Base Unit is configured via the REST API at `https://[base-unit-ip]:4003`:

```
PATCH /v2/configuration/features/digital-signage
{ "enabled": true, "url": "https://azleblanc.github.io/clickshare-signageV2/units/[unit-name]/" }
```

Authentication: HTTP Basic. Self-signed cert on Base Units — `--insecure` / `-SkipCertificateCheck` required for API calls on internal network.

## Future plans

**Management landing page** — a UI where marketing can view all units, see which images are loaded, and adjust `interval`/`transition` per unit without touching JSON directly. This is why the per-unit `images.json` abstraction matters — it's already the right data contract for a config UI to read and write.

**Content scheduling** — images with `startDate`/`endDate` metadata, filtered client-side by the slideshow JS.

**Image optimization workflow** — GitHub Action to auto-compress/resize uploads via ImageMagick before committing.

**Content approval workflow** — draft branch + PR review before content goes live.
